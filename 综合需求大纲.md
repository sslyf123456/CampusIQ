# 基于 RAG + 多智能体校园问答助手 —— 综合需求大纲

---

## 一、项目概述

### 1.1 项目组成

本项目由**两个后端子系统**与**一个前端应用**构成，通过内部 HTTP 接口协作，共同组成一个完整的校园智能问答平台：

| 子系统 | 定位 | 说明 |
|--------|------|------|
| **campus-service** | 校园后台支撑服务 | 提供用户鉴权、学生/课表/宿舍报修/奖助/通知的增删改查，以及供 AI 服务调用的内部只读接口。学生由管理员统一创建，不开放自主注册 |
| **ai-service** | AI 问答核心服务 | 多智能体协同系统，提供 RAG 知识库检索 + 多 Agent 意图路由 + 流式对话 |
| **前端** | 用户交互界面 | 涵盖登录注册、个人信息、AI 问答对话及校园数据展示等页面 |

### 1.2 协作关系

```
用户浏览器 / 前端
       │
       ▼
   Nginx (统一入口)
       │
       ├──── /api/campus/*  ──►  campus-service（校园数据）
       │
       └──── /api/ai/*      ──►  ai-service（AI 问答）
                                    │
                                    │ 内部 HTTP 调用（透传用户 JWT）
                                    ▼
                                campus-service 内部接口
```

- 用户在前端与 AI 助手对话，AI 助手通过调用校园**内部接口**获取实时个人数据（课表/宿舍报修/奖助/通知），生成自然语言回答
- AI 服务**不直连数据库**，所有业务数据通过 campus-service 暴露的内部接口获取
- 内部接口**透传用户 JWT**：ai-service 将前端用户的 JWT 原样传递给 campus-service，campus-service 自行解析 JWT 获取 student_id 并校验权限

### 1.3 核心闭环链路（总览）

```
注册 → 登录 → 进入系统 → 查看个人信息 / 课表 / 宿舍报修记录 / 奖助记录

AI 问答：用户提问 → 意图识别 → FAQ检索 / 路由Sub-Agent → 调校园内部接口 → 汇总回复

Agent路由：
  ScheduleAgent   → 查询课表
  RepairAgent     → 查询宿舍报修工单
  ScholarshipAgent→ 查询奖助记录
  NoticeAgent     → 检索校园通知
  FAQ兜底         → 向量知识库回答
```

### 1.4 优先级说明

| 级别 | 含义 |
|------|------|
| **P0** | 必须实现，Demo 演示要求完整可运行 |
| **P1** | 提供设计思路，开发时可保留接口但跳过具体实现 |

## 二、系统架构

### 2.1 服务职责

| 服务 | 核心职责 |
|------|----------|
| **campus-service** | Python FastAPI 实现，管理员创建学生账号、用户登录、JWT 鉴权、课表/宿舍报修/奖助/通知的增删改查、内部接口暴露 |
| **ai-service** | Python 实现，RAG 知识库向量检索、多智能体意图路由与编排（课表/宿舍报修/奖助/通知）、流式对话（SSE）、反问澄清、兜底回复 |

### 2.2 数据归属

| Schema | 归属服务 | 内容 |
|--------|----------|------|
| `campus` | campus-service | students、schedules、repair_orders、scholarship_records、campus_notices |
| `ai_campus` | ai-service | faq_embeddings（向量库）、conversations、messages |

### 2.3 基础设施

| 组件 | 用途 |
|------|------|
| **Nginx** | 统一入口，路径路由，CORS |
| **PostgreSQL + pgvector** | 关系数据 + 向量数据统一存储 |

### 2.4 前端

> 前端为用户与系统交互的唯一界面，需同时对接 campus-service 和 ai-service。前端采用 Vue 3 + TypeScript 开发，以下从用户视角描述前端应具备的页面、交互和体验要求。

#### 2.4.1 页面清单

> 前端所有页面（除登录/注册页外）均需登录才能访问。前端根据 JWT 中的角色（student / admin）灵活控制导航栏展示内容与页面入口，不同角色看到的页面和操作权限不同。

| 页面 | 学生可见 | 管理员可见 | 说明 |
|------|:---:|:---:|------|
| 登录页 | / | / | 学生/管理员统一登录入口（学号+密码），表单校验，未登录跳转到此页。不开放自主注册，学生账号由管理员创建 |
| 个人信息页 | / | / | 查看本人基本信息（姓名、学号、院系等），支持修改密码 |
| 课表查询页 | / | / | 学生：查看本周课程安排；管理员：维护课程数据（增删改） |
| 报修管理页 | / | / | 学生：发起宿舍报修、查看本人工单进度；管理员：查看全部工单、更新处理状态 |
| 奖助管理页 | / | / | 学生：查看本人奖助记录；管理员：增删改查全部奖助记录 |
| 校园通知页 | / | / | 学生：检索查看通知公告；管理员：发布/编辑/删除通知 |
| AI 问答对话主页 | / | / | 聊天气泡界面，SSE 流式输出，可折叠面板展示 Agent 推理过程，历史会话切换 |

#### 2.4.2 交互与体验要求

| 编号 | 要求 | 说明 |
|------|------|------|
| UX-01 | 表单校验 | 登录等表单需前端校验（学号格式、密码长度、必填项），提交前拦截明显错误 |
| UX-02 | 加载状态 | 所有异步请求需有加载指示（骨架屏、按钮 loading），避免用户感知卡死 |
| UX-03 | 空状态 | 空列表、无记录等场景需展示友好的空状态提示，而非空白页面 |
| UX-04 | 错误提示 | 后端返回错误时，以用户可理解的语言展示，网络异常时提示重试 |
| UX-05 | 操作反馈 | 关键操作成功/失败后需有明确反馈（Toast / 状态变更） |
| UX-06 | 登录态感知 | 除登录页外，所有页面均需登录才能访问。导航栏根据登录状态展示不同内容（未登录：仅登录页；已登录：学生/管理员对应导航菜单）；Token 过期时自动跳转登录页 |
| UX-07 | 权限感知 | 前端根据 JWT 中 role 字段（student / admin）动态渲染页面内容：学生仅看到自身数据的查询/宿舍报修入口，管理员看到管理操作入口（增删改按钮、全部数据视图）。未登录访问任何页面均跳转登录页 |

#### 2.4.3 AI 问答对话界面要求

| 编号 | 要求 | 说明 |
|------|------|------|
| AIX-01 | 流式输出 | 对话回复需以打字机效果逐字/逐句呈现，不等待完整回复后一次性展示 |
| AIX-02 | Agent 推理可见 | 中间推理阶段（路由决策 → Sub-Agent 执行中 → 结果汇总）以可折叠面板或状态标签形式展示 |
| AIX-03 | 消息气泡 | 用户消息与 AI 回复以聊天气泡形式区分，支持多轮对话滚动查看历史 |
| AIX-04 | 会话管理 | 支持新建会话、查看历史会话列表、切换会话 |
| AIX-05 | 输入状态 | AI 思考中时展示"正在输入..."动画，输入框暂不可用；完成后恢复 |
| AIX-06 | 错误兜底展示 | AI 调用校园接口失败时，仍以自然语言告知用户"暂时无法查询，请稍后重试" |

#### 2.4.4 跨服务交互要求

| 编号 | 要求 | 说明 |
|------|------|------|
| CX-01 | 统一鉴权 | 前端持同一 JWT 同时访问 campus-service 和 ai-service，用户无需分别登录 |
| CX-02 | 服务切换无感 | 用户在 AI 对话中查询课表时，前端不感知 ai-service 调用了 campus-service 内部接口，整个过程对用户透明 |

#### 2.4.5 权限路由设计

前端需实现路由守卫，确保登录和权限控制：

| 编号 | 要求 | 说明 |
|------|------|------|
| RX-01 | 路由守卫 | 除登录页外，所有路由均需校验 Token 存在且未过期，未登录自动重定向到登录页 |
| RX-02 | 角色差异化导航 | 学生角色（role=student）：导航栏展示个人信息、课表、宿舍报修、奖助、校园通知、AI 问答<br>管理员角色（role=admin）：导航栏展示个人信息、课表管理、宿舍报修管理、奖助管理、通知管理、AI 问答 |
| RX-03 | 页面内容按角色渲染 | 同一路由下，前端根据 JWT role 字段动态切换展示内容：学生端仅展示个人数据及查询操作，管理员端展示管理操作入口（增删改按钮、全量数据表格） |
| RX-04 | Token 过期处理 | 请求返回 401 时，清除本地 Token 并跳转登录页，提示"登录已过期，请重新登录" |
| RX-05 | 登录态持久化 | Token 存储在 localStorage，页面刷新后保持登录态，无需重复登录 |

### 3.4 多智能体调度模块

**归属服务**：ai-service

#### 3.4.1 多智能体架构

```
用户提问 → Master Agent（意图识别 + 路由）
                ├── 课表查询意图    → ScheduleAgent    → 调用 /internal/schedule    → 汇总
                ├── 宿舍报修查询意图    → RepairAgent      → 调用 /internal/repair       → 汇总
                ├── 奖助查询意图    → ScholarshipAgent → 调用 /internal/scholarship  → 汇总
                ├── 通知查询意图    → NoticeAgent      → 调用 /internal/notices      → 汇总
                └── 意图不明/超范围 → FAQ 检索兜底
```

#### 功能点清单

| 编号 | 功能点 | 优先级 | 说明 |
|------|--------|--------|------|
| AI-01 | Master 主 Agent 意图识别 | P0 | 分析用户问题，识别意图并路由到对应 Sub-Agent 或 FAQ 兜底 |
| AI-02 | 模糊问题反问澄清 | P0 | 意图模糊时主动反问用户确认需求，不强行猜测 |
| AI-03 | ScheduleAgent | P0 | 调用校园接口查询课表，透传 JWT 保证数据隔离 |
| AI-04 | RepairAgent | P0 | 调用校园接口查询宿舍报修进度，透传 JWT |
| AI-05 | ScholarshipAgent | P0 | 调用校园接口查询奖助记录，透传 JWT |
| AI-06 | NoticeAgent | P0 | 调用校园接口检索校园通知，透传 JWT |
| AI-07 | FAQ 兜底分支 | P0 | 无业务查询意图时走向量知识库回答 |

#### campus-service 需暴露的内部接口（供 ai-service 调用）

所有内部接口**复用 JWT 认证**：ai-service 将前端用户的 JWT 放入 `Authorization` 请求头透传给 campus-service，campus-service 解析 JWT 获取 `student_id`，数据隔离由 campus-service 保证。

| 接口 | HTTP 端点 | student_id 来源 | 用途 | 优先级 |
|------|-----------|----------------|------|--------|
| 课表查询 | `GET /internal/schedule` | JWT 解析 | 查询当前学生本周课表 | P0 |
| 宿舍报修查询 | `GET /internal/repair` | JWT 解析 | 查询当前学生宿舍报修工单 | P0 |
| 奖助查询 | `GET /internal/scholarship` | JWT 解析 | 查询当前学生奖助记录 | P0 |
| 通知检索 | `GET /internal/notices?keyword=...` | JWT 有效性校验 | 检索校园通知公告 | P0 |

**归属服务**：ai-service

| 编号 | 功能点 | 优先级 | 说明 |

| D-04 | 透传 JWT 数据隔离 | P0 | 会话绑定 student_id，用户 A 无法查询用户 B 数据 |

#### 闭环链路

```
[校园 FAQ 类问题]

[超出范围]
用户提问"明天天气怎么样？" → Master Agent 意图识别失败
    → FAQ 兜底检索 → 未匹配 → "抱歉，该问题超出我的服务范围，无法为您解答"
```

## 四、跨系统对接总览

### 4.1 接口调用清单

| 调用方 | 被调用方 | 接口 | 认证方式 | 用途 |
|--------|----------|------|----------|------|
| ai-service | campus-service | `GET /internal/schedule` | JWT（透传） | 查询学生课表，student_id 由 campus-service 解析 JWT 获取 |
| ai-service | campus-service | `GET /internal/repair` | JWT（透传） | 查询宿舍报修工单，student_id 由 campus-service 解析 JWT 获取 |
| ai-service | campus-service | `GET /internal/scholarship` | JWT（透传） | 查询奖助记录，student_id 由 campus-service 解析 JWT 获取 |
| ai-service | campus-service | `GET /internal/notices?keyword=...` | JWT（透传） | 检索校园通知，校验 JWT 有效性 |
| 前端 | campus-service | 全部 C 端 API | JWT Bearer Token | 用户登录、信息查询 |
| 前端 | ai-service | AI 对话 API | JWT Bearer Token | AI 问答对话，ai-service 将 JWT 透传给 campus-service |

### 4.2 数据隔离链路

```
学生A登录 → JWT(student_id=A) → 前端持 JWT 调用 AI 问答
    → ai-service 透传 JWT 调用 /internal/schedule
    → campus-service 解析 JWT → student_id=A → 查询 → 返回学生A的课表
    → 学生A无法通过 AI 或直接 API 获取学生B的数据（JWT 强制绑定身份）
```

### 4.3 异常降级链路

```
ai-service 调用 campus-service 内部接口
    → 正常返回 → 生成自然语言回答
    → 超时/错误 → 降级为 FAQ 检索 → 如 FAQ 也无匹配 → 友好提示用户稍后重试
    → ai-service 自身不崩溃
```

## 七、部署形态

采用**直接部署**方式，各服务以进程形式运行在本机或服务器上，无需容器化。

```
用户浏览器
     │
     ▼
  Nginx :80
     │
     ├──── /api/campus/*  ──►  campus-service  :8001
     │
     └──── /api/ai/*      ──►  ai-service      :8002
                                    │
                                    │ 内部 HTTP 调用
                                    ▼
                             campus-service :8001

  PostgreSQL + pgvector  :5432（本地数据库服务）
```

**启动顺序：**
1. 启动 PostgreSQL 数据库服务
2. 启动 campus-service（`uvicorn --host 0.0.0.0 --port 8001 app:app`）
3. 启动 ai-service（`uvicorn --host 0.0.0.0 --port 8002 app:app`）
4. 启动 Nginx（加载配置文件，反向代理两个后端服务）

**配置说明：**
- 各服务通过 `.env` 或配置文件管理数据库连接、端口、JWT 密钥等敏感信息，不提交 git
- Nginx 配置反向代理规则，统一处理 CORS，前端访问 `/api/campus/*` 和 `/api/ai/*` 分别转发到对应服务

## 八、附录

### 8.1 关键业务规则速查

| 规则 | 说明 |
|------|------|
| 学生/管理员权限隔离 | 学生仅可操作本人数据（课表、宿舍报修、奖助查询），管理员可进行增删改查管理操作，JWT 强制绑定身份 |
| campus-service 写权限 | 学生侧仅可发起宿舍报修（增），其余写操作（学生/课程/奖助/通知的增删改）仅管理员可执行 |
| AI 数据获取 | 仅通过内部 HTTP 接口，透传用户 JWT 认证，不直连数据库 |
| 演示数据 | 基础演示数据（学生/课程/奖助/通知）可预置，支持通过管理后台动态维护 |
| 兜底策略 | 校园接口失败→降级 FAQ；FAQ 无匹配→统一兜底回复 |

### 8.2 状态速查

| 实体 | 状态值 | 说明 |
|------|--------|------|
| 宿舍报修工单 | `pending` / `processing` / `completed` | 待处理 / 处理中 / 已完成 |
| 奖助申请 | `approved` / `pending` / `rejected` | 已发放 / 审核中 / 已拒绝 |
| AI 会话 | `active` / `closed` | 活跃 / 已关闭 |
